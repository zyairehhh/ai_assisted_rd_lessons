# L01 课前讲义：从已有代码仓开始

## 1. 本课要解决什么

本课训练的是最基础但最重要的研发闭环：

```text
读任务 -> 读代码 -> 定范围 -> 修改 -> 测试 -> 修失败 -> 总结
```

很多 AI 辅助研发失败，不是因为模型不会写代码，而是因为任务边界、上下文和验证方式没有给清楚。

## 2. 建议工作流

### Step 1: 自己先读任务

先不要把任务直接丢给 AI。你需要先知道：

- 要修什么。
- 哪些目录可能相关。
- 验证命令是什么。
- 哪些文件不应该改。

### Step 2: 让 agent 先分析

建议 prompt：

```text
请先不要修改代码。请阅读 task.md、README.md 和测试失败信息，说明你认为可能相关的文件、根因假设和验证计划。
```

### Step 3: 限制修改范围

建议 prompt：

```text
请只修改 src/minidb 中和 LIMIT 处理相关的代码。不要修改测试来绕过问题，不要引入新依赖。
```

### Step 4: 小步修改和验证

修改后运行：

```bash
./validators/validate.sh
```

如果失败，让 agent 基于失败输出继续定位。

## 3. 本课常见误区

| 误区 | 问题 |
|---|---|
| 直接让 AI 修复所有失败 | 容易扩大修改范围 |
| 不看测试输出 | 不知道验证标准 |
| 让 AI 改测试绕过失败 | 失去训练意义 |
| 不记录上下文 | 无法复盘 agent 为什么做错 |
| 只看代码能跑 | 缺少修复说明和风险记录 |

## 4. 本课产出

你需要提交：

- `context.md`
- `validation_report.md`
- 修复后的源码
- 如有必要，补充测试
