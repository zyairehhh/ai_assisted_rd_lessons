# Coding Agent 行为规范

本仓库是 AI 辅助研发培训仓。Coding Agent 在本仓库中工作时，应遵守以下规则。

## 一、通用规则

1. 先阅读当前目录的 `README.md` 和 `lesson.yaml`，再制定计划。
2. 修改练习时，默认只在当前课程目录内工作。
3. 不要跨课程修改材料，除非任务明确要求。
4. 不要生成或泄露标准答案。
5. 不要引入不必要的外部依赖。
6. 不要把工具配置、token、内部地址写入仓库。
7. 修改代码或文档后，尽量运行当前课程的验证脚本。

## 二、练习工作流

处理 `exercise/` 中的任务时，应遵循：

1. 阅读 `exercise/README.md` 和 `exercise/task.md`。
2. 明确任务目标、限制范围和验证方式。
3. 维护或更新 `context.md`。
4. 先分析和计划，再修改代码或文档。
5. 小步修改，小步验证。
6. 在输出中说明修改内容、验证结果和剩余风险。

## 三、上下文记录要求

每次练习都应维护 `context.md`，记录：

- 本次任务是什么。
- 给 Coding Agent 提供了哪些上下文。
- 明确限制不要改哪些内容。
- 如何验证结果。
- Agent 的哪些判断被人工纠正。
- 哪些内容后续适合沉淀成 skill 或知识库。

模板见 `shared/templates/context.template.md`。

## 四、代码修改要求

1. 遵循当前练习项目已有代码风格。
2. 不做与任务无关的重构。
3. 不随意扩大修改范围。
4. 新增行为应补充测试。
5. 修复 bug 应尽量补充回归测试。
6. 修改构建脚本或依赖时，应说明原因。

## 五、Review 类任务要求

处理代码检视任务时：

1. 输出必须引用证据，例如文件、函数、日志或测试结果。
2. 区分确定问题、疑似问题和建议。
3. 标注严重级别。
4. 明确哪些结论需要人工确认。
5. 不要把风格偏好包装成确定缺陷。

## 六、测试类任务要求

处理测试设计或测试代码生成任务时：

1. 先生成测试矩阵，再生成测试代码。
2. 覆盖正常路径、边界条件、错误路径和回归场景。
3. 尽量学习已有测试风格。
4. 不要只为覆盖率写无意义测试。
5. 说明测试能发现什么问题，以及发现不了什么问题。

## 七、Debug 类任务要求

处理问题定位任务时：

1. 先整理事实，不要直接猜根因。
2. 建立假设列表。
3. 为每个假设标注证据和验证方法。
4. 保留排查过程，形成 `debug_journal.md`。
5. 最终输出根因、修复建议和回归方案。

## 八、Skill 和知识库任务要求

构建 skill 时，应明确：

- 适用场景。
- 输入材料。
- 执行步骤。
- 输出格式。
- 检查清单。
- 正例和反例。
- 验证方法。

构建知识库时，应明确：

- 来源材料。
- 脱敏方式。
- 标签和索引。
- 引用格式。
- 过期或失效条件。
- 检索问题集。
